# Product OS v1

## Purpose
A repeatable system for deciding what to build, in what order, for whom, and with what level of automation, so 20–30 products can be produced without strategic drift.

## Core Rule
Do not build products. Build repeatable revenue machines.

---

## 1. Product Selection Filter

Before any product moves forward, it must pass these filters:

### 1.1 Pain
- Is the pain frequent?
- Is the pain expensive?
- Is the pain operationally annoying?
- Is the pain already budgeted for in some form?

### 1.2 Buyer
- Can we name the first buyer precisely?
- Do they already spend money to patch this problem?
- Is the buyer reachable through content, network, or direct outreach?

### 1.3 Execution leverage
- Can we build an MVP quickly?
- Can it reuse our existing design/components?
- Can it be automated heavily?
- Can it be sold before perfect polish?

### 1.4 Strategic fit
- Does it increase revenue?
- Does it save D. time?
- Does it compound into a broader system?
- Does it create reusable infrastructure?

Anything weak on all four gets parked.

---

## 2. Product Categories

Every product should be labeled as one of these:

### 2.1 Revenue driver
Directly sells or captures demand.
Examples:
- AI Grading Assistant
- Meeting Output Engine
- niche SaaS tools

### 2.2 Distribution driver
Creates audience or inbound demand.
Examples:
- Content Factory
- social content engines
- SEO / content surfaces

### 2.3 Internal leverage tool
Improves speed, quality, or automation for the portfolio.
Examples:
- Design OS
- Product OS
- dashboard systems
- agent workflows

### 2.4 Infrastructure layer
Supports multiple future products.
Examples:
- auth
- billing shell
- component system
- AI routing layer
- integrations layer

Rule: prefer products that hit multiple categories.

Each product moves through these phases:

### Phase 1 — Extraction
- pain extraction

### Phase 3 — Visual architecture
- screen list
- block hierarchy
- key actions

### Phase 4 — Functional architecture

### Phase 5 — Build

### Phase 6 — Monetization
- pricing
- first offer

### Phase 7 — Systemization
- templates

## 4. Product Scoring System

Score each idea 1–5 on:
- pain intensity
- buyer clarity
- speed to MVP
- reuse across portfolio
- automation potential
- low infra cost
- monetization speed
- differentiation
- ease of distribution
- strategic relevance

### Score interpretation
- 42–50 = build now
- 34–41 = strong candidate
- 26–33 = backlog / refine
- <26 = ignore for now

---

## 5. MVP Rules

### 5.1 MVP must prove one thing
Not everything.
Examples:
- can it produce post-meeting output automatically?
- can teachers pay to reduce grading burden?
- can content convert into inbound?

### 5.2 MVP must avoid
- premature teams
- premium infra
- overbuilt permissions
- full settings galaxies
- unnecessary mobile apps first
- enterprise fantasies too early

### 5.3 MVP should favor
- one sharp use case
- one buyer
- one clear workflow
- one repeatable proof point

---

## 6. Build Economics

### 6.1 Preferred cost structure
- local-first when possible
- free or near-free inference first
- premium APIs only where quality matters materially
- reusable backend patterns
- minimal hosting footprint

### 6.2 Red flags
- product only works with expensive inference
- product depends on too many third-party APIs
- margins die at scale
- support burden grows linearly

---

## 7. Interface and Automation Doctrine

Every product should answer:
- what is zero-touch?
- what is one-touch?
- what requires review?
- what stays hidden until asked?

If a workflow repeats, systemize it.
If a user must think too much, redesign it.
If the product needs explaining, sharpen the primary action.

## 8. Product Archetype Templates

### 8.1 Capture → Transform → Route
For tools like the Meeting Output Engine.

### 8.2 Generate → Review → Publish
For content systems.

### 8.3 Input → Score → Return
For grading/evaluation products.

### 8.4 Detect → Recommend → Execute
For internal ops / automation tools.

### 8.5 Dashboard → Exception → Action
For management surfaces.

These archetypes should reuse the Design OS screen library.

---

## 9. Distribution Doctrine

A product without distribution is a hobby.
For each product define:
- top acquisition channel
- fastest proof channel
- outbound angle
- content angle
- referral or compounding loop

Preferred channels by default:
- content-led inbound
- direct founder outreach
- community/operator circles
- owned dashboard/content surfaces

---

## 10. Decision Rules

### Build now when:
- pain is obvious
- buyer is reachable
- MVP can be built fast
- infra cost is low
- distribution path exists

### Delay when:
- product is interesting but buyer unclear
- requires too much custom infra too early
- depends on expensive AI from day one
- would distract from a nearer revenue opportunity

### Kill when:
- no painful problem
- no buyer urgency
- no repeatability
- no strategic leverage
